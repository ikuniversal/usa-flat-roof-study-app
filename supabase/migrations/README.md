# Migrations

## 0001 (initial schema)
The initial schema lives in `content/07_supabase_seed.sql`, which was applied
when the Supabase project was first set up. It defines all 17 tables, RLS,
the `handle_new_user` trigger, and a sample seed (1 part / 1 chapter /
3 sections / 1 quiz / 1 glossary term / 1 comparison table). There is no
`0001_*.sql` file in this directory — the seed file IS the starting point.

## 0002_profiles_self_select.sql
Phase 4.5 fix #3 — adds `public.is_admin(uuid)` SECURITY DEFINER helper and
rewrites every admin RLS policy that previously did `EXISTS (SELECT 1 FROM
profiles WHERE id = auth.uid() AND role = 'admin')` to use it instead. The
old form recurses on profiles policies and triggers `42P17 infinite
recursion in policy`, which used to crash the post-login dashboards.
**Applied via MCP. Live.**

### How to apply pending chunks

**Option A — Supabase SQL Editor (no local CLI needed).**
Open the project at https://supabase.com/dashboard/project/incnssapeuouhsglkfno,
click **SQL Editor**, and paste each file in order. Each chunk is wrapped
in `BEGIN; ... COMMIT;` and is idempotent.

**Option B — psql via the Supabase database URL.**
```bash
for f in supabase/migrations/0003_chunk_*.sql; do
  psql "$SUPABASE_DB_URL" -f "$f" || break
done
```

**Option C — `npm run import:content`** (the original 1571-line TS script
that uses the service-role key over the Supabase JS client). This requires
`.env.local` to contain `SUPABASE_SERVICE_ROLE_KEY` and a network path to
the project. The chunked SQL files exist as a fallback for environments
where direct egress is blocked (e.g. iPad sandboxes).

### Regenerating the chunks

```bash
npx tsx scripts/generate-import-sql.ts
```

Reads `content/01..06_*.json`, writes `supabase/migrations/0003_chunk_*.sql`.
Re-run safe — each chunk is overwritten in place.
